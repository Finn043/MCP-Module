1. MCP là gì?

Model Context Protocol (MCP) là một giao thức chuẩn hóa để cung cấp context và tools cho AI models (như Claude) mà không cần viết integration code phức tạp. MCP chuyển gánh nặng định nghĩa và thực thi tools từ ứng dụng của bạn sang các MCP server chuyên biệt.

Ở cốt lõi, MCP định nghĩa một kiến trúc chuẩn:

  • Một MCP Client (trong ứng dụng của bạn) kết nối đến một hoặc nhiều MCP Server
  • Mỗi MCP Server chứa tools, prompts, và resources
  • Mỗi MCP Server đóng vai trò là interface đến một dịch vụ bên ngoài

2. Vấn đề MCP giải quyết

Hãy tưởng tượng bạn xây dựng một giao diện chat nơi người dùng hỏi Claude về dữ liệu GitHub của họ. GitHub có rất nhiều chức năng — repositories, pull requests, issues, projects, và nhiều hơn nữa.

Không có MCP — Vấn đề N x M

Bạn cần tạo một số lượng khổng lồ tool schemas và functions — viết, kiểm thử, và bảo trì tất cả integration code đó. Nếu có N ứng dụng cần tích hợp với M services, bạn cần N x M custom integrations.

Có MCP — Giải pháp N + M

Một MCP Server cho GitHub xử lý tất cả. Nó bao bọc nhiều chức năng và expose chúng dưới dạng một tập tools chuẩn hóa. Ứng dụng của bạn kết nối đến MCP server thay vì tự triển khai mọi thứ. Kết quả: chỉ cần N + M implementations (N clients + M servers).

Giống như cổng USB-C. Trước đây, mỗi thiết bị cần một loại cáp riêng. USB-C chuẩn hóa kết nối — mọi thiết bị dùng chung một cổng. MCP làm điều tương tự cho kết nối giữa AI và dịch vụ bên ngoài.

3. Kiến trúc tổng thể

MCP có kiến trúc 4 thành phần chính:

Host → MCP Client → MCP Server → External API
H
Host — Ứng dụng chính mà người dùng tương tác. Ví dụ: Claude Desktop, VS Code, hoặc một web app. Host chứa MCP Client và gửi yêu cầu thay mặt người dùng.
C
MCP Client — Cầu nối giao tiếp giữa Host và MCP Server. Client xử lý giao thức, transport, và message exchange. Client có thể kết nối nhiều server cùng lúc.
S
MCP Server — Interface đến dịch vụ bên ngoài. Server expose tools, resources, và prompts theo format chuẩn. Ví dụ: GitHub MCP Server, Database MCP Server.
E
External API — Dịch vụ thực tế mà MCP Server kết nối đến. GitHub API, database, file system, etc.

4. Luồng truy vấn hoàn chỉnh

Khi người dùng hỏi "Tôi có những repositories nào?", đây là luồng xử lý:

1
User Query — Người dùng gửi câu hỏi đến Host (ứng dụng của bạn).
2
Tool Discovery — Host cần biết có tools nào sẵn sàng để Claude sử dụng.
3
List Tools Exchange — Host hỏi MCP Client cho danh sách tools; Client gửi ListToolsRequest đến MCP Server, nhận ListToolsResult.
4
Claude Request — Host gửi câu hỏi + danh sách tools đến Claude.
5
Tool Use Decision — Claude quyết định cần gọi một tool để trả lời câu hỏi.
6
Tool Execution — Host yêu cầu MCP Client chạy tool Claude đã chọn. Client gửi CallToolRequest đến MCP Server.
7
External API Call — MCP Server gọi GitHub API thực tế.
8
Results Flow Back — API phản hồi; dữ liệu chảy ngược qua MCP Server thành CallToolResult.
9
Tool Result to Claude — Host gửi kết quả tool quay lại Claude.
10
Final Response — Claude tổng hợp câu trả lời cuối cùng bằng dữ liệu nhận được.
11
User Gets Answer — Host hiển thị câu trả lời cho người dùng.

5. MCP Servers — Ba Primitives

MCP Server expose ba loại primitives theo cách chuẩn hóa:

PrimitiveĐược điều khiển bởiMục đíchVí dụ
ToolsClaude (AI model)Cho Claude khả năng mới để tự chủ thực thiChạy code, tính toán, tìm kiếm
ResourcesỨng dụng (app code)Fetch dữ liệu vào ứng dụng cho UI hoặc contextDanh sách autocomplete, nội dung tài liệu
PromptsNgười dùngTemplate hướng dẫn sẵn để người dùng kích hoạtSlash commands, workflow buttons

Ví dụ: GitHub MCP Server chứa tools như get_repos() và kết nối trực tiếp đến GitHub API. Ứng dụng của bạn giao tiếp với MCP server, server xử lý tất cả chi tiết đặc thù của GitHub.

6. MCP Clients

MCP Client là cầu nối giao tiếp giữa Host và MCP Servers. Nó xử lý message exchange và chi tiết giao thức.

Các loại message chính

RequestResponseMục đích
ListToolsRequestListToolsResultHỏi "bạn cung cấp tools gì?"
CallToolRequestCallToolResultYêu cầu server chạy một tool
ReadResourceRequestReadResourceResultFetch dữ liệu resource theo URI
ListPromptsRequestListPromptsResultLiệt kê các prompt templates
GetPromptRequestGetPromptResultLấy một prompt cụ thể với arguments

7. Transport Agnostic

MCP là transport-agnostic — client và server có thể giao tiếp qua nhiều giao thức khác nhau:

TransportMô tảTrường hợp sử dụng
Standard I/O (stdio)Client và server chạy trên cùng một máy, giao tiếp qua stdin/stdoutPhát triển local, phổ biến nhất
HTTP / StreamableHTTPKết nối đến remote server qua HTTPServer public, triển khai production
WebSocketsKết nối song song hai chiều real-timeỨng dụng web real-time

Lưu ý: MCP là giao thức bidirectional — cả client và server đều có thể khởi tạo giao tiếp. Điều này quan trọng khi chọn transport, vì HTTP có hạn chế server-to-client communication.
