1. Tại sao cần HTTP transport?

stdio transport rất tuyệt nhưng có một giới hạn nghiêm trọng: chỉ hoạt động trên cùng máy. Trong thực tế, bạn cần:

  • Remote MCP servers — Server chạy trên máy khác, client kết nối qua Internet
  • Public MCP servers — Server công khai, bất kỳ ai cũng có thể kết nối
  • Production deployment — Triển khai trên cloud infrastructure

StreamableHTTP transport cho phép MCP client kết nối đến remote servers qua HTTP — mở ra khả năng deploy MCP server ở bất kỳ đâu.

2. Vấn đề cốt lõi của HTTP

HTTP được thiết kế cho client-initiated communication. Điều này tạo ra vấn đề nghiêm trọng cho MCP — vì MCP cần bidirectional communication.

Client CÓ THỂ:
→ Gửi requests đến server (server có URL)

Server CÓ THỂ:
← Response cho client requests

Server KHÔNG THỂ:
→ Initiate request đến client (không có client URL!)

Tại sao đây là vấn đề?

Nhớ lại bài trước: MCP là bidirectional. Server cần gửi requests đến client (sampling, list roots) và notifications (progress, logging). HTTP chuẩn không hỗ trợ điều này!

3. StreamableHTTP — Không phải SSE cũ!

Quan trọng: Spec đã thay đổi!

MCP specification đã thay đổi transport từ SSE cũ sang StreamableHTTP. Đừng nhầm lẫn:

  • SSE cũ (deprecated) — Yêu cầu endpoint riêng /sse, phức tạp hơn, mỗi message cần endpoint riêng
  • StreamableHTTP (hiện tại) — Đơn giản hơn, tích hợp SSE vào chính HTTP response. Đây là spec hiện tại.

StreamableHTTP giải quyết vấn đề bằng cách sử dụng Server-Sent Events (SSE) bên trong HTTP response — không cần endpoint riêng.

4. StreamableHTTP Workaround — SSE trong HTTP Response

StreamableHTTP sử dụng SSE (Server-Sent Events) bên trong HTTP response để cho phép server gửi messages về client. Đây là workaround thông minh cho HTTP limitation.

Initialize Flow:

1
Client gửi POST Initialize Request — Standard HTTP POST đến server URL.
2
Server phản hồi với Initialize Result + mcp-session-id header — Response chứa session ID để tracking.
3
Client gửi POST Initialized Notification với session ID — Xác nhận đã khởi tạo.
Client ── POST /mcp ──→ Server
Initialize Request

Client ←── 200 OK + SSE ── Server
Initialize Result + mcp-session-id header

Client ── POST /mcp ──→ Server
Initialized Notification (với session ID)

5. Session ID

Server trả về mcp-session-id trong Initialize response header. Session ID này:

  • Uniquely identifies client connection
  • Phải include trong TẤT CẢ requests sau initialization
  • Cho phép server tracking state của từng client
  • Bị mất khi stateless_http=True
// Initialize Response Headers
HTTP/1.1 200 OK
Content-Type: text/event-stream
mcp-session-id: abc123-def456-ghi789

6. Hai Config Flags

StreamableHTTP có hai configuration flags quan trọng ảnh hưởng đến functionality:

FlagMặc địnhKhi True
stateless_httpFalseKhông tracking session, không server-to-client requests. Mất: sampling, progress, logging.
json_responseFalseDisable streaming. Chỉ trả final JSON result. Không có intermediate messages.

Thiết lập trong code

mcp = FastMCP(
    "mcp-server",
    stateless_http=True,   # Mặc định False
    json_response=True,    # Mặc định False
)

mcp.run(transport="streamable-http")

7. Tính năng nào bị ảnh hưởng?

Khi stateless_http=True, các tính năng cần server-to-client communication bị mất:

Tính năngYêu cầustateless_http=True
SamplingServer → Client requestBROKEN
Progress notificationsServer → Client messageBROKEN
Logging messagesServer → Client messageBROKEN
Resource subscriptionsServer → Client messageBROKEN
Tool resultsClient request → Server responseVẫn hoạt động
List toolsClient request → Server responseVẫn hoạt động

Lưu ý quan trọng: ctx.info() và ctx.report_progress() sẽ silently fail khi stateless — client không bao giờ nhận được chúng. Không có error, chỉ là messages bị mất.
