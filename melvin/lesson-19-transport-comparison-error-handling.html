1. So sánh Transport

MCP hỗ trợ nhiều transport methods. Mỗi cái có trade-offs riêng:

Feature stdio StreamableHTTP (stateful) StreamableHTTP (stateless)
Remote access ❌ Same machine only ✅ Remote ✅ Remote
Bidirectional ✅ Perfect ✅ Via SSE ❌ No
Sampling ✅ ✅ ❌
Progress/Logging ✅ ✅ ❌
Horizontal scaling ❌ Single process ⚠️ Sticky sessions ✅ Load balancer OK
Setup complexity Low Medium Medium
Use case Dev/Testing Production single server Production at scale

2. Khi nào dùng Transport nào?

Decision flowchart để chọn transport phù hợp:

Cần kết nối từ máy khác?
↓ Không → stdio (Dev/Testing)
↓ Có → Cần horizontal scaling?
↓ Không → StreamableHTTP stateful
↓ Có → Cần sampling/notifications?
↓ Có → ⚠️ Cần sticky sessions
↓ Không → StreamableHTTP stateless

3. Error Handling trong MCP

MCP sử dụng JSON-RPC 2.0 error codes:

Error CodeTênÝ nghĩa
-32700Parse errorJSON không hợp lệ
-32600Invalid RequestRequest structure sai
-32601Method not foundMethod không tồn tại
-32602Invalid paramsTham số không hợp lệ
-32603Internal errorLỗi nội bộ server
Tool Execution Errors: Khi tool gặp lỗi, response có flag isError: true. Khác với JSON-RPC errors, đây là lỗi trong quá trình thực thi tool, không phải lỗi protocol.

Ví dụ error response:

{
  "jsonrpc": "2.0",
  "id": 1,
  "error": {
    "code": -32601,
    "message": "Method not found",
    "data": { "method": "tools/unknown_method" }
  }
}
// Tool execution error
{
  "jsonrpc": "2.0",
  "id": 2,
  "result": {
    "content": [
      {"type": "text", "text": "Error: File not found at /path/to/file"}
    ],
    "isError": true
  }
}

4. Best Practices

1
Validate at boundaries — Kiểm tra inputs ngay khi nhận được từ client. Đừng trust data từ bên ngoài.
2
Return meaningful errors — Thay vì generic "Error", cung cấp context: file nào, path nào, expected format gì.
3
Test với production transport — Behavior khác nhau giữa stdio và HTTP. Luôn test với transport bạn sẽ deploy.
4
Handle transport errors gracefully — Connection drops, timeouts, network issues. Client nên có retry logic.

5. Testing Advice

⚠️ QUAN TRỌNG: Luôn test với production transport của bạn. Behavior khác nhau đáng kể giữa:
  • stdio vs HTTP — stdio hỗ trợ full bidirectional, HTTP có limitations
  • stateful vs stateless — Notifications, sampling, progress chỉ hoạt động với stateful
  • json_response=True — Streaming disabled, chỉ nhận final result

Tool chạy hoàn hảo với stdio nhưng có thể "im lặng" khi deploy với stateless_http=True. Đừng phát hiện vấn đề sau khi deploy!

Common Error Scenarios

Nhấn vào mỗi error scenario để xem JSON-RPC response tương ứng.

-32700: Parse Error — Gửi JSON không hợp lệ 
// Request
{ "jsonrpc": "2.0", "method": "tools/list", INVALID }

// Response
{
  "jsonrpc": "2.0",
  "id": null,
  "error": {
    "code": -32700,
    "message": "Parse error"
  }
}
-32601: Method Not Found — Gọi method không tồn tại 
// Request
{ "jsonrpc": "2.0", "method": "tools/unknown", "id": 1 }

// Response
{
  "jsonrpc": "2.0",
  "id": 1,
  "error": {
    "code": -32601,
    "message": "Method not found"
  }
}
isError: true — Tool execution error 
// Tool call: convert_video("nonexistent.mp4")
{
  "jsonrpc": "2.0",
  "id": 3,
  "result": {
    "content": [
      {
        "type": "text",
        "text": "Error: Input file not found: nonexistent.mp4"
      }
    ],
    "isError": true
  }
}
-32602: Invalid Params — Tham số sai kiểu 
// Request: add("abc", "xyz") thay vì add(1, 3)
{
  "jsonrpc": "2.0",
  "id": 4,
  "error": {
    "code": -32602,
    "message": "Invalid params",
    "data": {
      "field": "a",
      "expected": "integer",
      "received": "string"
    }
  }
}
