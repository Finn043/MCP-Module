1. Tổng quan về JSON Messages trong MCP

Mọi giao tiếp trong MCP đều diễn ra thông qua JSON messages, tuân theo chuẩn JSON-RPC 2.0. Khi client cần gọi một tool trên server, client gửi một JSON message; server xử lý và phản hồi bằng một JSON message khác.

Định dạng chung của MCP JSON message

{
  "jsonrpc": "2.0",
  "id": 1,
  "method": "tools/call",
  "params": {
    "name": "add",
    "arguments": { "a": 5, "b": 3 }
  }
}

Mỗi message có jsonrpc version, id để tracking, method để xác định hành động, và params chứa dữ liệu.

Có 2 loại chính:

Request-Result — Có response. Gửi request, chờ result. Ví dụ: CallToolRequest → CallToolResult.

Notification — Không có response. One-way message. Gửi xong không mong đợi phản hồi. Ví dụ: Progress Notification.

2. Request-Result Pairs

Những message này luôn đi theo cặp. Mỗi request phải có một result tương ứng. Đây là backbone của MCP communication.

3. Notifications — Tin nhắn một chiều

Notifications là messages không mong đợi response. Chúng đơn giản chỉ thông báo về một event nào đó đã xảy ra.

NotificationMục đíchAi gửi?
Progress NotificationCập nhật tiến độ của tác vụ dàiServer → Client
Logging Message NotificationGửi log messages (info, warning, error)Server → Client
Tool List Changed NotificationThông báo danh sách tools đã thay đổiServer → Client
Resource Updated NotificationResource đã được cập nhậtServer → Client
Initialized NotificationClient xác nhận đã khởi tạo xongClient → Server

Đặc điểm của Notification

Notification không có id — vì không cần tracking response. Format đơn giản hơn request:

{
  "jsonrpc": "2.0",
  "method": "notifications/progress",
  "params": {
    "progress": 50,
    "total": 100
  }
}

4. MCP là Bidirectional — Cả hai phía đều có thể chủ động

Đây là insight quan trọng nhất về MCP: CẢ client VÀ server đều có thể gửi request và notification.

Client → Server:
CallToolRequest → Server xử lý → CallToolResult

Server → Client:
CreateMessageRequest → Client gọi LLM → CreateMessageResult
Hãy tưởng tượng MCP như một cuộc điện thoại — cả hai bên đều có thể nói và nghe. Client không chỉ "hỏi" và server không chỉ "trả lời". Server cũng có thể chủ động yêu cầu client làm việc (như sampling).

MCP specification chia messages theo ai gửi:

  • Client messages — Requests client gửi đến server (tool calls, list requests) và notifications client gửi
  • Server messages — Requests server gửi đến client (sampling, list roots) và notifications server broadcast

5. Tại sao hiểu message types quan trọng khi chọn Transport?

MCP hỗ trợ nhiều transport methods (stdio, HTTP, ...). Mỗi transport có khả năng khác nhau về việc hỗ trợ bidirectional communication.

Câu hỏi then chốt khi chọn transport

Transport này có hỗ trợ bidirectional messaging không?

  • stdio — Có. Hoàn hảo. Cả hai phía gửi/nhận thoải mái qua stdin/stdout.
  • HTTP — Giới hạn. Client dễ gửi request đến server, nhưng server khó initiate request đến client.

Khi bạn hiểu rằng MCP cần bidirectional communication, bạn sẽ hiểu tại sao:

1
stdio hoạt động hoàn hảo — stdin/stdout cho phép cả hai chiều.
2
HTTP cần workaround (SSE) — vì HTTP vốn là request-response, không phải bidirectional.
3
Một số MCP features bị mất khi dùng transport không hỗ trợ đầy đủ bidirectional.

Nối tiếp: Bài tiếp theo sẽ đi sâu vào stdio transport — transport đơn giản nhất và hỗ trợ bidirectional hoàn hảo.
