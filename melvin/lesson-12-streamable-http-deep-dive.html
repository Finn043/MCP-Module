1. Dual SSE Connections — Hai kết nối đồng thời

Khi client gọi một tool qua StreamableHTTP, hệ thống tạo HAI SSE connections cùng lúc. Mỗi connection có vai trò khác nhau:

Primary SSE (GET request)
Client GET /mcp → Server
Stays open indefinitely — server-initiated messages

Tool-specific SSE (POST response)
Client POST /mcp → Server
Đóng sau khi tool result sent

Primary SSE Connection (GET)

  • Client mở bằng GET request đến server
  • Connection stays open indefinitely
  • Server dùng channel này để gửi server-initiated messages
  • Cho phép: sampling requests, progress notifications, resource subscriptions

Tool-specific SSE Connection (POST response)

  • Tạo khi client gửi POST request để gọi tool
  • Server response là một SSE stream
  • Cho phép: logging messages, tool results, intermediate progress
  • Đóng sau khi tool result đã được gửi xong

2. Message Routing — Message đi qua channel nào?

Khác loại message được route qua khác SSE connection. Hiểu routing là key để debug StreamableHTTP:

Loại MessageĐi qua channel nàoLý do
Progress notificationsPrimary SSE (GET)Server-initiated, không gắn với tool call cụ thể
Logging messagesTool-specific SSE (POST)Gắn với tool call đang thực thi
Tool resultsTool-specific SSE (POST)Phản hồi cho tool call cụ thể
Server-initiated requests (sampling)Primary SSE (GET)Server cần initiate bất cứ lúc nào
Hãy tưởng tượng hai đường điện thoại: một đường "hotline" luôn mở (Primary SSE) cho server gọi cho client bất cứ lúc nào, và một đường "task line" (Tool SSE) chỉ mở khi cần thực hiện một công việc cụ thể rồi đóng lại sau.

3. stateless_http=True — Tại sao cần?

Vấn đề: Horizontal Scaling

Khi MCP server trở nên phổ biến, bạn cần chạy nhiều server instances phía sau load balancer. Nhưng đây là vấn đề:

Client A ── GET SSE ──→ Load Balancer → Server 1
Client A ── POST ──→ Load Balancer → Server 2

COORDINATION PROBLEM!
GET SSE và POST có thể đi đến khác server instances

Nếu POST đến Server 2, nhưng Primary SSE mở với Server 1, Server 2 không thể gửi message qua SSE connection của Server 1.

Solution: stateless_http=True

Bỏ hoàn toàn server-to-client communication:

  • Client không nhận session ID — server không tracking ai
  • GET SSE pathway không còn — không server-initiated messages
  • Mỗi POST request là independent — không cần coordination
  • Nhưng mất: Sampling, progress, logging, subscriptions

Trade-off

ĐượcMất
Horizontal scalingSampling (server-initiated)
Load balancer compatibleProgress notifications
Không cần sticky sessionsLogging messages
Mỗi request independentResource subscriptions
Initialized notifications tracking

4. json_response=True — Disable Streaming

json_response đơn giản hơn — chỉ tắt streaming cho POST request responses.

SettingResponse FormatIntermediate Messages
json_response=False (mặc định)SSE streamProgress, logs, rồi final result
json_response=TruePlain JSONChỉ final tool result

Khi json_response=True: không có intermediate progress messages, không có log statements trong khi tool chạy. Client chỉ nhận một JSON response duy nhất — kết quả cuối cùng.

mcp = FastMCP(
    "mcp-server",
    stateless_http=True,   # Bỏ server-to-client
    json_response=True,    # Tắt streaming
)
# ctx.info() và ctx.report_progress() sẽ SILENTLY FAIL!

5. Decision Guide — Khi nào dùng gì?

Quyết định chọn transport và config

Kịch bảnTransportConfig
Development / TestingstdioMặc định — full features
Production, single server, cần full featuresStreamableHTTPstateless=False, json_response=False
Production, load balancer, horizontal scalingStreamableHTTPstateless=True
Production, cần simple JSON responsesStreamableHTTPjson_response=True

Quy tắc vàng

Luôn test với production transport! Nếu bạn develop với stdio nhưng deploy với HTTP, behavior có thể khác đáng kể. Catch issues during development, not after deployment.

Warning

Với stateless_http=True và json_response=True, các lệnh ctx.info() và ctx.report_progress() sẽ silently fail — client không bao giờ nhận được chúng. Không có error, chỉ là messages biến mất.
