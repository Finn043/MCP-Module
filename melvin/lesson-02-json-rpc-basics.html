1. JSON-RPC 2.0 là gì?

JSON-RPC 2.0 là một giao thức Remote Procedure Call (RPC) sử dụng JSON để mã hóa dữ liệu. Nó cho phép một chương trình gửi yêu cầu thực thi một hàm trên một chương trình khác — có thể ở cùng máy hoặc qua mạng.

MCP được xây dựng hoàn toàn trên JSON-RPC 2.0. Mọi tin nhắn MCP — từ initialize đến tools/call — đều là tin nhắn JSON-RPC 2.0 hợp lệ. Hiểu JSON-RPC là điều kiện tiên quyết để hiểu MCP.

JSON-RPC giống như gửi một bức thư yêu cầu ai đó làm giùm việc. Bạn ghi rõ tên việc (method), thông số (params), và số tham chiếu (id). Người nhận làm xong rồi gửi lại kết quả cùng số tham chiếu để bạn biết kết quả nào thuộc về yêu cầu nào.

2. Request Format

Một Request trong JSON-RPC 2.0 phải chứa các trường sau:

TrườngKiểuMô tả
jsonrpcstringLuôn là "2.0"
idnumber | stringĐịnh danh request, dùng để match response
methodstringTên phương thức cần gọi
paramsobject | arrayTham số truyền cho phương thức (có thể bỏ qua)

Ví dụ thực tế — Client MCP yêu cầu danh sách tools:

{
  "jsonrpc": "2.0",
  "id": 1,
  "method": "tools/list",
  "params": {}
}

3. Response Format

Server trả về một trong hai dạng response:

Response thành công (Success)

Chứa trường result cùng id khớp với request:

{
  "jsonrpc": "2.0",
  "id": 1,
  "result": {
    "tools": [
      { "name": "read_doc", "description": "Đọc tài liệu" }
    ]
  }
}

Response lỗi (Error)

Chứa trường error thay vì result:

{
  "jsonrpc": "2.0",
  "id": 1,
  "error": {
    "code": -32601,
    "message": "Method not found"
  }
}

4. Error Codes

JSON-RPC 2.0 định nghĩa các error code chuẩn:

CodeTênMô tả
-32700Parse errorJSON không hợp lệ, server không parse được
-32600Invalid RequestRequest thiếu trường bắt buộc hoặc sai format
-32601Method not foundMethod không tồn tại trên server
-32602Invalid paramsTham số không hợp lệ
-32603Internal errorLỗi nội bộ phía server

Các error code từ -32000 đến -32099 được dành cho server-defined errors. MCP sử dụng khoảng này cho các lỗi riêng.

5. Notification

Notification giống Request nhưng KHÔNG có trường id. Vì không có id, server không trả về response nào.

Notification dùng cho các thông báo một chiều (fire-and-forget). Ví dụ trong MCP: sau khi nhận Initialize Result, client gửi notifications/initialized để xác nhận — không mong đợi response.

{
  "jsonrpc": "2.0",
  "method": "notifications/initialized"
}

Lưu ý: params trong notification là tùy chọn. Không có id = không có response.

6. Batch Request

JSON-RPC 2.0 cho phép gửi nhiều request cùng lúc trong một JSON array:

[
  { "jsonrpc": "2.0", "id": 1, "method": "tools/list", "params": {} },
  { "jsonrpc": "2.0", "id": 2, "method": "resources/list", "params": {} },
  { "jsonrpc": "2.0", "id": 3, "method": "prompts/list", "params": {} }
]

Server sẽ trả về một array các response theo cùng thứ tự (match theo id). Điều này giúp giảm số lượng round-trip khi cần gọi nhiều method.

7. Kết nối với MCP

Mọi tin nhắn MCP đều là tin nhắn JSON-RPC 2.0. Khi bạn thấy tools/list, tools/call, initialize, hay notifications/initialized — tất cả đều tuân theo format JSON-RPC 2.0.

Hiểu JSON-RPC giúp bạn:

  • Debug bất kỳ giao tiếp MCP nào bằng cách đọc JSON thô
  • Hiểu tại sao một số message có response và một số thì không
  • Phân biệt request lỗi do format sai hay do logic
  • Tự viết MCP client/server bằng bất kỳ ngôn ngữ nào
MCP Client --- JSON-RPC 2.0 Request ---> MCP Server
MCP Client <-- JSON-RPC 2.0 Response --- MCP Server
MCP Client --- JSON-RPC 2.0 Notification ---> MCP Server (Không có response)

JSON-RPC 2.0 Message Builder

Chọn loại tin nhắn, điền các trường, và xem JSON-RPC 2.0 message được tạo ra.

Kết quả:

// Chọn loại tin nhắn và nhấn "Tạo tin nhắn"

Hướng dẫn:

  • Request — Cần id, method, và params
  • Response (Success) — Cần id và result
  • Response (Error) — Cần id và error (code + message)
  • Notification — Chỉ cần method và params, không có id

Bài tập 1: Hoàn thành JSON-RPC Request ⬜ Chưa làm

Điền các trường còn thiếu để tạo một JSON-RPC Request hợp lệ. Request này yêu cầu danh sách tools từ MCP server.

💡 Gợi ý: JSON-RPC Request cần 4 trường: jsonrpc (luôn là "2.0"), id (số hoặc chuỗi), method (tên method), và params (tham số).
Giải pháp:
{ "jsonrpc": "2.0", "id": 1, "method": "tools/list", "params": {} }
Bài tập 2: Phân loại tin nhắn ⬜ Chưa làm

Chọn loại tin nhắn JSON-RPC phù hợp cho mỗi ví dụ dưới đây.

💡 Gợi ý: Request có method và id. Notification có method nhưng không có id. Success Response có result và id. Error Response có error và id.
Message A:
{"jsonrpc":"2.0","id":1,"method":"initialize","params":{}}
Message B:
{"jsonrpc":"2.0","method":"notifications/initialized","params":{}}

Message A thuộc loại:

Request
Notification
Error Response

Message B thuộc loại:

Request
Notification
Error Response
