1. stdio Transport hoạt động như thế nào?

Trong stdio transport, client khởi chạy server như một subprocess và giao tiếp qua standard input/output streams.

Client ── ghi vào stdin ──→ Server (subprocess)

Client ←── đọc từ stdout ── Server (subprocess)

Quá trình hoạt động:

1
Client khởi chạy server — ví dụ uv run server.py như một subprocess.
2
Client ghi JSON messages vào server's stdin — gửi requests, notifications.
3
Server ghi JSON messages ra stdout — gửi responses, server-initiated requests.
4
Bất kỳ phía nào cũng có thể gửi message bất kỳ lúc nào — true bidirectional.

Chỉ hoạt động khi client và server chạy trên cùng một máy. Đây là giới hạn duy nhất của stdio.

2. Line-delimited JSON

Mỗi message được gửi trên một dòng riêng biệt (newline-delimited). Format phổ biến nhất:

Format chuẩn

Content-Length: 123\r\n
\r\n
{"jsonrpc":"2.0","id":1,"method":"initialize","params":{...}}

Hoặc đơn giản hơn — mỗi line là một JSON object hoàn chỉnh, phân tách bởi newline character.

Hãy tưởng tượng stdin và stdout như hai ống nước nối giữa client và server. Client đổ JSON vào ống stdin, server nhận được. Server đổ JSON vào ống stdout, client nhận được. Hai ống này độc lập — cả hai đều có thể chảy cùng lúc.

3. 3-message Handshake qua stdio

Mọi kết nối MCP đều bắt đầu bằng 3-message handshake. Qua stdio, chỉ cần đúng 2 channels:

Client ── stdin ──→ Server
1. Initialize Request

Client ←── stdout ── Server
2. Initialize Result

Client ── stdin ──→ Server
3. Initialized Notification

Chỉ sau khi handshake hoàn tất mới có thể gửi các requests khác (tool calls, prompt listings, etc.).

4. Bốn Communication Patterns

Với stdio, cả 4 pattern giao tiếp đều hoạt động hoàn hảo:

HướngLoạiChannelVí dụ
Client → ServerRequestClient ghi vào stdinCallToolRequest
Server → ClientResponseServer ghi ra stdoutCallToolResult
Server → ClientRequestServer ghi ra stdoutCreateMessageRequest (sampling)
Client → ServerResponseClient ghi vào stdinCreateMessageResult

Bidirectional hoàn hảo!

stdio cho phép cả 4 pattern — đây là lý do nó được gọi là "ideal" transport. Không có giới hạn nào về hướng giao tiếp.

5. stderr cho Debug

Server có một channel thứ ba: stderr. Channel này hoàn toàn riêng biệt và không ảnh hưởng đến protocol communication (stdin/stdout).

Các role của stderr

  • Debug logs — Server ghi log/debug info ra stderr
  • Error messages — Thông báo lỗi không liên quan đến MCP protocol
  • Diagnostics — Client có thể đọc stderr riêng để hiển thị debug info
Client ── stdin ──→ Server (MCP protocol)

Client ←── stdout ── Server (MCP protocol)

Client ←── stderr ── Server (debug/logs)

6. stdio là "Ideal" Transport

Tại sao stdio được recommended cho development?

  • Full bidirectional — Cả 4 communication patterns đều hoạt động
  • Đơn giản — Chỉ cần 2 channels (stdin + stdout)
  • Không cần network — Không lo firewall, port, CORS
  • Dễ debug — stderr riêng cho logs
  • Khởi tạo đơn giản — mcp.run(transport="stdio")

Nhưng có giới hạn!

Chỉ hoạt động trên cùng máy. Nếu cần remote server (production deployment), phải dùng transport khác như StreamableHTTP.

# Server code đơn giản
from mcp.server.fastmcp import FastMCP

mcp = FastMCP(name="My Server")

@mcp.tool()
def add(a: int, b: int) -> int:
    return a + b

if __name__ == "__main__":
    mcp.run(transport="stdio")
